From Ruhoh to Hexo

（2021年3月 Hexo博客及Stellar主题定制）

Kevin Kelly在《技术元素》中说：创作者，如艺术家、音乐家、摄影师、工匠、演员、动画师、设计师、视频制作者，或者作家——换之，也就是任何创作艺术作品的人 —— 只需拥有1000名铁杆粉丝便能糊口。
我从不奢望个人博客会成为一个多么火爆的站点，实际上它就是我简简单单用来回顾和分享所学知识的地方。不过，努力撰写有质量的，原创的文章，来吸引至少 1000 名铁杆粉丝，应该是每个独立 blogger 的基本追求。

我在2021年初把博客迁移到了Ruhoh，用了一段时间之后，开始感觉到各种不满意的地方:
(1)页面生成速度太慢。我的整个站点总共写了20来篇文章，在生成的时候花了将近半分钟时间。
(2)定制性有限。由于作者一开始就把 Ruhoh 定位在 language agnostic 的目标，因此在采用的模板语言也选用了最平台无关的 mustache。但 mustache 的语法能力实在有限，只支持简单的逻辑判断。像传参、循环之类的特性都不支持。比如我想实现类似 octopress 这样的 tag 插件，直接用 mustache 是不可能的，只能动 ruhoh 的 ruby 源码，而这样又和作者的设计目标相反了。
(3)太小众。用 Ruhoh 的人太少，插件相应也少，不像 Jekyll 或 octopress 那样有丰富的扩展。
(4)不跨平台。Ruhoh 在 Windows 下总会遇到一些奇怪的问题而无法运行。
(5)更新缓慢。Ruhoh 的已经半年没有重大更新了，到现在标签依然还停留在 next-release 阶段。最致命的一个问题是在一个 issue 页面里，作者跟我们承认他自己平常也很少使用 Ruhoh 。可想而知，如果一个产品连开发者自己都不用的话，很难保证这个产品会做得好。
出于以上种种原因，我决定弃用 Ruhoh ，改用其他的博客系统。

后来在Github上看到很多开发者在使用 hexo ，一个基于 node.js 的博客系统。本来我就对 node.js 非常感兴趣，于是立马去 hexo 的主页了解更多它的特性。在看了它的文档之后，我立马把它列为我的 first choice 。与 Ruhoh 相比，hexo 有以下几个优点：
（1）页面生成速度超快。我的20来篇文章只用了不到两秒就生成完毕。
（2）使用 ejs 作为模板驱动，本身保留了 Javascript 语法的特性，所以定制性很强。
（3）用户数量多，插件丰富。也许是因为作者是台北同胞的原因，项目的issue页可以看到很多中文的issue，加上作者的项目管理非常合理，不仅有比较完善的项目主页和文档，也有很好的插件wiki，然后又有 node.js 的 npm 社区支持，感觉非常 promising 。
（4）跨平台。javascript跨平台特性相当不错，在 Windows 下跑 hexo 不存在太大问题。
（5）更新频繁。作者自己也用 hexo，可以看看他的博客。

后续计划
上面主要描述了自己对博客系统的一些使用感受，感觉都有点不务正业了。接下来就想赶紧回到正轨，开始我的毕业论文的工作，再同时把新博客用上，不断地总结我学到的新东西。当然，现在的博客还有一些不完美的地方，接下来也想试着去完成：

（1）hexo 会在使用过程中产生 db.json 文件，记录整个站点中的所有文章的信息。理论上可以使用 ajax 和 jquery 这个文件来检索，实现一个比 google 更好的本站搜索引擎。
（2）优化我的 mathjax 插件，在 development 模式下加载本地的 mathjax 脚本，在 production 模式下加载 CDN 上的版本（这个计划被迫放弃了。Mathjax太大了，试图本地加载它会导致hexo的服务器启动半天，实在划不来）。
（3）标签栏的划分不明确，需要花时间去找下源代码，然后针对性地修改。
